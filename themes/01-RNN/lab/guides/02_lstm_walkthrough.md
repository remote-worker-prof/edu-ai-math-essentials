## TODO 1-2: кумулятивная сумма и метки
### Что нужно понять
- исходный `x` всё ещё имеет форму `(N, T, 1)`;
- сначала удобно убрать последнюю ось признака, чтобы получить массив формы `(N, T)`;
- затем по временной оси считается `np.cumsum(...)`;
- целевая метка нужна на каждом шаге, поэтому в конце ось признака надо вернуть.

### Логика решения
- `cumsum` должен иметь форму `(N, T)`;
- `y` должен иметь форму `(N, T, 1)`;
- в этой лабораторной класс `1` ставится, если текущая накопленная сумма строго больше `0`, иначе `0`.

### Ожидаемые результаты
- `X.shape == (N, T, 1)`
- `cumsum.shape == (N, T)`
- `y.shape == (N, T, 1)`

### Типичные ошибки
- считать `cumsum` по неправильной оси;
- забыть вернуть последнюю ось признака;
- получить `y.shape == (N, T)` вместо `(N, T, 1)`.

### Self-check
- Почему для `y` нельзя оставить форму `(N, T)`?
  Потому что модель после `Dense(1)` возвращает `(batch, T, 1)`, и целевая форма должна совпадать.
- Что происходит на границе `sum == 0`?
  Для этой лабораторной класс остаётся `0`, потому что правило — строгое `> 0`.

## TODO 3-4: сборка модели
### Что нужно понять
- формат `many-to-many` требует выхода на каждом шаге;
- значит рекуррентный слой обязан возвращать всю последовательность состояний;
- именно поэтому нужен `LSTM(..., return_sequences=True)`.

### Логика решения
- поставить `LSTM` на 16-32 нейрона;
- обязательно включить `return_sequences=True`;
- завершить модель `Dense(1, activation='sigmoid')`;
- компилировать как бинарную классификацию.

### Ожидаемые результаты
- `model.output_shape == (None, T, 1)` для нужного `T`;
- встроенная мини-проверка сравнения с `y_train.shape[1:]` проходит.

### Типичные ошибки
- забыли `return_sequences=True`;
- модель возвращает `(batch, units)`;
- перепутали смысл `Dense(1)` и ожидали `(batch, 1)`.

### Self-check
- Почему без `return_sequences=True` ЛР2 концептуально не решается?
  Потому что модель выдаст один вектор на всю последовательность вместо ответа на каждом шаге.

## TODO 6-7: `preds` и `sequence_accuracy`
### Что нужно понять
- `model.predict` по-прежнему отдаёт вероятности, а не классы;
- сначала нужен `preds` после порога `0.5`;
- потом нужно проверить, совпала ли вся последовательность целиком.

### Логика решения
- получить бинарные `preds` той же формы, что и `y_test`;
- поэлементно сравнить `preds == y_test`;
- для `sequence_accuracy` проверить, что совпали все шаги и все признаки одного объекта;
- после этого усреднить по объектам.

### Worked example
Пусть:

```text
y_true = [[1], [0], [1], [1]]
preds  = [[1], [0], [1], [0]]
```

Тогда:
- `token_accuracy = 3 / 4 = 0.75`
- `sequence_accuracy = 0`, потому что в последовательности есть одна ошибка

### Ручная таблица
| Шаг | `y_true` | `preds` | Совпадение |
|---|---|---|---|
| 1 | `1` | `1` | да |
| 2 | `0` | `0` | да |
| 3 | `1` | `1` | да |
| 4 | `1` | `0` | нет |

### Типичные ошибки
- считать `sequence_accuracy` как среднее по шагам, а не по последовательностям;
- забыть порог `0.5` перед вычислением `preds`;
- сравнивать массивы несовместимых форм.

### Self-check
- Почему `token_accuracy` может быть высокой, а `sequence_accuracy` низкой?
  Потому что одна ошибка внутри последовательности не сильно портит локальную метрику, но обнуляет строгую последовательную.
