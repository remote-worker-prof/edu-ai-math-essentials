# Beginner Guide: Lab 2 `LSTM` many-to-many

## Главная идея этой лабораторной
Теперь модель должна отвечать не один раз в конце, а на каждом шаге времени.

В этой постановке:
- вход — последовательность `-1` и `+1`;
- на шаге `t` нужно предсказать знак кумулятивной суммы;
- формат задачи — `many-to-many`.

## Интуиция задачи
Модель идёт слева направо и всё время отвечает на вопрос:

```text
Если остановиться прямо сейчас, сумма уже положительная или нет?
```

Поэтому каждая новая метка зависит не только от текущего элемента, но и от всей истории до него.

## Ручной пример
Пусть:

```text
x = [+1, -1, +1, +1]
```

Тогда кумулятивная сумма по шагам:

```text
t=1:  1
t=2:  0
t=3:  1
t=4:  2
```

Если в лабораторной метка задаётся как “сумма положительна”, то по шагам получится:

```text
y = [1, 0, 1, 1]
```

Это уже не один итоговый ответ, а целая последовательность ответов.

## Что изменилось относительно ЛР1
| Было в ЛР1 | Стало в ЛР2 |
|---|---|
| один ответ на последовательность | ответ на каждом шаге |
| `y.shape == (N,)` | `y.shape == (N, T, 1)` |
| достаточно финального состояния | нужны состояния на всех шагах |
| метрика `accuracy` по объектам | `token_accuracy` + `sequence_accuracy` |
| `SimpleRNN` в формате `many-to-one` | `LSTM` в формате `many-to-many` |

Это не “чуть более сложная та же задача”, а уже другой тип выхода модели.

## Почему нужен `return_sequences=True`
Если `return_sequences=False`, рекуррентный слой вернёт только последнее состояние:

```text
(batch, units)
```

Это подходит для `many-to-one`, но не подходит для `many-to-many`.

Если `return_sequences=True`, слой возвращает состояния на каждом шаге:

```text
(batch, time, units)
```

И тогда можно делать предсказание на каждом шаге.

## Визуальная интуиция для `return_sequences=True`
Полезно читать это буквально через формы:

```text
вход                 -> (batch, time, features)
LSTM без sequences   -> (batch, units)
LSTM с sequences     -> (batch, time, units)
Dense(1) после этого -> (batch, time, 1)
```

То есть параметр `return_sequences=True` не “улучшает модель магически”, а заставляет её вернуть именно тот тип выхода, который нужен постановке `many-to-many`.

## Почему здесь используется `LSTM`, а не `SimpleRNN`
В ЛР1 хватало простой памяти.

Здесь модель должна аккуратно переносить информацию по времени на всех шагах, поэтому `LSTM` удобна как более устойчивая архитектура:
- есть отдельный канал памяти;
- есть ворота, регулирующие поток информации;
- на практике обучение обычно стабильнее на более длинных зависимостях.

## Важная анти-путаница: `token_accuracy` не равна tokenization
В этой лабораторной слово `token` появляется только в названии метрики `token_accuracy`.

Здесь это означает:
- один элемент последовательности;
- один шаг времени;
- одну локальную метку.

Это **не** NLP tokenization.

Здесь нет:
- словаря;
- токенизатора;
- текстовых токенов;
- `PAD/SOS/EOS`.

То есть `token_accuracy` в ЛР2 лучше читать как:

```text
“доля правильных ответов по шагам времени”
```

## Почему `sequence_accuracy` строже
`token_accuracy` смотрит на каждый шаг отдельно.

`sequence_accuracy` спрашивает:

```text
Была ли вся последовательность предсказана без единой ошибки?
```

Из-за этого возможна типичная ситуация:
- почти все шаги предсказываются верно;
- но в каждой последовательности есть 1 ошибка;
- тогда `token_accuracy` высокая, а `sequence_accuracy` заметно ниже.

## Три ручных примера: высокая `token_accuracy`, низкая `sequence_accuracy`
Пусть длина последовательности равна `4`.

Пример 1:

```text
y_true = [1, 0, 1, 1]
preds  = [1, 0, 1, 0]
```

- `token_accuracy = 3 / 4 = 0.75`
- `sequence_accuracy = 0`, потому что есть одна ошибка

Пример 2:

```text
y_true = [0, 0, 1, 1]
preds  = [0, 0, 1, 1]
```

- `token_accuracy = 1.00`
- `sequence_accuracy = 1`

Пример 3:

```text
y_true = [1, 1, 0, 0]
preds  = [1, 1, 0, 1]
```

- `token_accuracy = 0.75`
- `sequence_accuracy = 0`

Именно поэтому для ЛР2 нельзя смотреть только на одну метрику.

## Как читать формы в ЛР2
- `X`: `(N, T, 1)`
- `y`: `(N, T, 1)`
- выход модели: `(batch, T, 1)`

Визуальный признак `many-to-many`: у `y` и у выхода есть временная ось `T`.

## Как читать графики
- Если `token_accuracy` растёт, модель учится на локальных шагах.
- Если `sequence_accuracy` сильно отстаёт, модель всё ещё допускает точечные ошибки.
- Если `val_loss` начинает расти при хорошем train, это сигнал возможного переобучения.

## Отдельно про `token_accuracy` и tokenization
В ЛР2 термин `token` используется в очень узком техническом смысле метрики.

Здесь:
- `token_accuracy` = доля правильных шагов времени;
- `token` = один локальный элемент последовательности;
- tokenization текста отсутствует полностью.

То есть в этой лабораторной нет:
- словаря;
- `Tokenizer`;
- текстовых токенов;
- специальных токенов вроде `PAD/SOS/EOS`.

Если в голове возникает ассоциация с NLP, её лучше сразу отключить: она понадобится позже, но не здесь.

## Порядок диагностики
1. Проверить, что `y` имеет форму `(N, T, 1)`.
2. Проверить правило построения кумулятивной суммы.
3. Проверить `return_sequences=True`.
4. Сравнить `token_accuracy` и `sequence_accuracy`, а не смотреть только на одну метрику.

## Что должно остаться после ЛР2
- отличие `many-to-many` от `many-to-one`;
- смысл `return_sequences=True`;
- разница между локальной и строгой последовательной метрикой;
- понимание, что `token_accuracy` здесь — это шаги времени, а не токенизация текста.
