# Debugging Playbook: `02-Attention / ЛР01`

## С чего начинать, если attention-лабораторная не сходится
Порядок проверки должен быть жёстким:
1. Сначала убедиться, что работает plain `seq2seq`-часть.
2. Потом проверить формы attention-тензоров.
3. И только после этого смотреть на heatmap и интерпретацию.

Если начать с картинки внимания слишком рано, легко лечить не причину, а симптом.

## Шаг 1. Проверка данных
Сначала руками проверьте один пример:
- `encoder_input` содержит исходную последовательность и `PAD`;
- `decoder_input` начинается с `SOS`;
- `decoder_target` заканчивается на `EOS`;
- формы совпадают с `(N, 10)`, `(N, 11)`, `(N, 11, 1)`.

Типичные сбои:
- забыли `SOS`;
- забыли `EOS`;
- перепутали `decoder_input` и `decoder_target`;
- потеряли последнюю ось у `decoder_target`.

## Шаг 2. Проверка plain `seq2seq`-логики
Даже attention-модель ломается, если сломан базовый pipeline из `01-RNN / ЛР03`.

Проверьте:
- используется ли `Embedding(mask_zero=True)`;
- корректен ли сдвиг decoder;
- обучается ли модель через `[encoder_input, decoder_input]`;
- соответствует ли выход `(batch, T_out, VOCAB_SIZE)`.

Если уже на этом уровне формы не сходятся, attention ещё не при чём.

## Шаг 3. Проверка encoder и decoder
Нужно отдельно подтвердить:
- encoder `GRU` имеет `return_sequences=True` и `return_state=True`;
- decoder `GRU` имеет `return_sequences=True` и `return_state=True`;
- decoder запускается с `initial_state=encoder_state`.

Частая ошибка:
- encoder возвращает только `state`, и тогда attention просто не к чему прикладываться.

## Шаг 4. Проверка attention-вызова
Правильный вызов в этой лабораторной:

```python
context, attention_scores = attn(
    [decoder_outputs, encoder_outputs],
    return_attention_scores=True,
)
```

Проверить нужно три вещи:
- `decoder_outputs` идут первыми и играют роль `query`;
- `encoder_outputs` идут вторыми и играют роль `key/value`;
- `attention_scores` действительно возвращаются.

## Шаг 5. Проверка форм analysis-тензоров
Ожидаемый публичный контракт:
- `encoder_outputs`: `(N, T_in, H)`
- `decoder_outputs`: `(N, T_out, H)`
- `context`: `(N, T_out, H)`
- `attention_scores`: `(N, T_out, T_in)`

Если хотя бы одна из этих форм не сходится, дальше интерпретация уже бессмысленна.

## Шаг 6. Проверка метрик
Ищите не только абсолютные значения, но и согласованность:
- `token_accuracy` должна быть заметно выше случайного уровня;
- `exact_match` обычно ниже `token_accuracy`;
- обе метрики должны расти по мере обучения.

Если `token_accuracy` растёт, а `exact_match` почти нет:
- проверьте decoder shift;
- проверьте число эпох;
- разберите один пример вручную по токенам.

## Шаг 7. Проверка heatmap
Смотреть на heatmap имеет смысл только после всех предыдущих шагов.

Перед чтением карты:
- уберите PAD-строки и PAD-столбцы;
- подпишите оси;
- выберите один понятный тестовый пример.

Для reverse-задачи хороший паттерн:
- фокус идёт примерно по антидиагонали;
- нет ощущения полного хаоса;
- внимание сдвигается по входу, а не застревает в одной колонке.

## Частые симптомы и быстрые проверки
- Симптом: `attention_scores` неожиданной формы.
  Проверка: сравнить фактическую форму с `(batch, T_out, T_in)` и перепроверить порядок аргументов `Attention(...)`.
- Симптом: heatmap пустая или бессмысленная.
  Проверка: убедиться, что отображаются только значимые позиции без `PAD`.
- Симптом: модель почти не лучше baseline.
  Проверка: убедиться, что задача действительно длинная (`ENC_LEN=10`, длина `4..10`), иначе преимущество attention сглаживается.
- Симптом: обучение падает с ошибкой shapes.
  Проверка: начать с `decoder_target.shape` и `model.output_shape`.

## Как пользоваться solution notebook без самообмана
Правильный порядок:
1. Собственная попытка.
2. Проверка walkthrough.
3. Локализация конкретного шага.
4. И только потом сверка с solution notebook по одному блоку.

Неправильный порядок:
- открыть готовое решение сразу;
- вставить весь код;
- получить working notebook без понимания, где именно появляется `context` и зачем нужен `attention_scores`.
