# Foundations Guide: Self-Study Debugging Playbook

## Зачем нужен этот файл
Если вы проходите курс без преподавателя, главная проблема обычно не в формуле, а в том, что непонятно, где именно сломалась работа:
- в данных;
- в форме тензоров;
- в модели;
- в masking и метриках;
- в самом состоянии notebook.

Этот playbook даёт единый порядок восстановления для `RNN`, `seq2seq` и `attention`.

## Самый надёжный режим работы с notebook
1. Запускать ячейки сверху вниз.
2. Не перепрыгивать к нижним разделам, пока не пройдены мини-проверки выше.
3. После правок в середине notebook использовать `Restart & Run All`.
4. Если вывод внезапно “странный”, сначала подозревать состояние kernel, а не математику.

## Как читать traceback
Правильный порядок:
1. Смотреть на последнюю строку traceback.
2. Определить тип ошибки.
3. Вернуться к строке, на которую указывает сообщение.
4. Проверить форму и тип объектов, попавших в эту строку.

Самые частые классы:
- `NameError` — ячейка выше не запускалась или переменная названа иначе.
- `ValueError` — чаще всего рассыпались формы.
- `AssertionError` — сработала встроенная мини-проверка и уже сузила место поиска.

## Shape-First Debugging
Перед всем остальным полезно проверить:

```python
print(X.shape)
print(y.shape)
print(model.output_shape)
```

Для `seq2seq`:

```python
print(encoder_input.shape)
print(decoder_input.shape)
print(decoder_target.shape)
print(model.output_shape)
```

Для attention:

```python
print(encoder_outputs.shape)
print(attention_scores.shape)
```

Если формы не сходятся с контрактом, дальнейшая диагностика почти всегда преждевременна.

## Где искать ошибку
### 1. Данные
Признаки:
- метрики около случайного уровня;
- ручной пример и `y` не совпадают;
- тестовая мини-проверка данных падает.

Что делать:
- взять один пример;
- посчитать expected label вручную;
- проверить форму batch и временной оси.

### 2. Модель
Признаки:
- `model.output_shape` неожиданная;
- recurrent-слой возвращает не ту размерность;
- `fit(...)` падает на несовпадении форм.

Что делать:
- для `many-to-one` проверить, не включили ли лишний sequence-output;
- для `many-to-many` проверить `return_sequences=True`;
- для `seq2seq` проверить оба входа и последний `Dense(V)`.

### 3. Метрики и постобработка
Признаки:
- built-in `accuracy` не совпадает с ручным пересчётом;
- `token_accuracy` хорошая, а `exact_match` неожиданно слабый;
- внимание выглядит шумно, хотя модель как будто учится.

Что делать:
- проверить порог или `argmax`;
- убедиться, что `PAD` не участвует в строгой метрике;
- проверить, не мешают ли padded позиции чтению heatmap.

## Минимальный маршрут восстановления
1. `Restart & Run All`.
2. Проверить shapes.
3. Проверить один ручной пример.
4. Проверить ближайшую мини-проверку из notebook.
5. Открыть соответствующий walkthrough.
6. Только потом сверяться с solution notebook.

## Как пользоваться walkthrough и solution
Правильный порядок:
1. Своя попытка.
2. Walkthrough только для текущего шага.
3. Повторная попытка.
4. И только потом solution notebook.

Неправильный порядок:
- сразу открыть решение;
- скопировать блок;
- получить зелёную ячейку без понимания причины.

## Особые подсказки для seq2seq и attention
- Если `token_accuracy` приемлемая, а `exact_match` низкая, сначала проверяйте decoder shift.
- Если attention-heatmap не читается, сначала уберите `PAD`, а не меняйте модель вслепую.
- Если модель “учится”, но выдаёт бессмысленные последовательности, проверьте vocabulary, `SOS/EOS` и masking.

## Что должно остаться
- привычка начинать с shapes;
- понятный порядок: данные -> модель -> метрики -> только потом решение;
- уверенность, что большинство ошибок локализуются быстрее через контракт данных, чем через длинные догадки.
