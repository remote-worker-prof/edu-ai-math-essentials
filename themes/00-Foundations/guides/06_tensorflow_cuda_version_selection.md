# How to Think About TensorFlow and CUDA Versions for Local GPU

Этот guide нужен не для первого безопасного запуска, а для понимания:
- какой локальный GPU-path курс считает нормальным;
- когда достаточно `tensorflow[and-cuda]`;
- когда уже не стоит бесконечно править notebook, а нужно думать про совместимость GPU-стека.

Главная идея:
- для большинства студентов нужен один `blessed path`;
- для сильных студентов полезно понимать, как читать official TensorFlow docs и tested build configurations;
- курс не ожидает, что вы с первого запуска станете "системным интегратором CUDA".

## Один Рекомендуемый Путь Для Курса
Если вам нужен локальный GPU, в курсе считается нормальным только такой сценарий:
- `Linux + NVIDIA`;
- или `Windows + WSL2 + Ubuntu`, если вы работаете на Windows.

Для обычного student-flow рекомендуемый путь такой:
1. создать `.venv`;
2. установить зависимости курса;
3. поверх них поставить GPU-вариант TensorFlow:

```bash
python3 -m pip install --upgrade 'tensorflow[and-cuda]>=2.16,<2.20'
```

4. проверить:

```bash
python3 - <<'PY'
import tensorflow as tf
print("built_with_cuda =", tf.test.is_built_with_cuda())
print("gpus =", tf.config.list_physical_devices("GPU"))
PY
```

5. если GPU виден, использовать `RUNTIME_MODE = "local-gpu"`.

Для локального CPU вам не нужен GPU-стек. Штатный CPU-path:
- обычный `tensorflow`;
- или уже подготовленное course-окружение без локального GPU-ускорения.

## Что Официально Говорит TensorFlow
По официальной pip-документации TensorFlow:
- для локального GPU на `Linux` и `WSL2` рекомендуется `tensorflow[and-cuda]`;
- проверка делается через `tf.config.list_physical_devices("GPU")`;
- native Windows GPU для современных версий TensorFlow не является основным путём;
- для старых версий и нестандартных сценариев нужно смотреть official tables с tested build configurations.

Практический перевод на язык курса:
- сначала идите по blessed path;
- не начинайте с ручного подбора `CUDA` и `cuDNN`;
- только если blessed path не сработал, переходите к advanced compatibility reasoning.

## Ментальная Модель Зависимостей
Чтобы не воспринимать GPU-путь как магию, полезно видеть 4 слоя:

1. `NVIDIA driver` на хосте.
2. `TensorFlow wheel` в вашей `.venv`.
3. pip-installed `nvidia-*` runtime packages, которые подтягиваются через `tensorflow[and-cuda]`.
4. Архитектура самой видеокарты.

Из этого следует важная мысль:
- `GPU виден` и `все notebook'и стабильно работают` — это не одно и то же.

Возможны три разных состояния:
- `nvidia-smi` не работает -> проблема ещё до TensorFlow;
- TensorFlow не видит GPU -> это setup-проблема;
- TensorFlow видит GPU, но реальные kernels падают -> это уже compatibility-проблема.

## Как Выбирать Версии На Практике
Начинайте не с вопроса "какие версии поставить", а с вопроса "какой сценарий мне нужен":

### Сценарий 1. Хочу просто пройти лабораторную
Не тратьте время на локальный GPU, если он не нужен.

Спокойные варианты:
- `RUNTIME_MODE = "auto"`
- `RUNTIME_MODE = "local-cpu"`
- `RUNTIME_MODE = "colab-gpu"`
- `RUNTIME_MODE = "kaggle-gpu"`

### Сценарий 2. Хочу именно локальный GPU
Сначала идите только по blessed path:
- `Linux` или `WSL2`;
- `tensorflow[and-cuda]`;
- проверка через `tf.config.list_physical_devices("GPU")`.

Если это сработало и notebook'и тоже работают, на этом подбор версий заканчивается.

### Сценарий 3. GPU уже виден, но обучение падает
Это уже не повод бесконечно менять модельный код.

Сначала зафиксируйте, что:
- `tf.config.list_physical_devices("GPU")` не пустой;
- проблема проявляется во время реальной GPU-работы;
- на CPU тот же notebook ведёт себя нормально.

Тогда почти наверняка вы имеете дело не с ошибкой студента в модели, а с совместимостью локального GPU-стека.

## Когда Не Надо Вручную Подбирать CUDA и cuDNN
Не учите себя вручную собирать матрицу
`TensorFlow x Python x CUDA x cuDNN` в таких случаях:
- вы используете обычный локальный курс без source-build;
- вы работаете с актуальным pip-путём TensorFlow;
- вам нужен не эксперимент, а просто рабочий учебный запуск.

Для такого режима ручной подбор версий обычно только усложняет жизнь.

## Когда Уже Нужно Смотреть Official Tables
Идти в official TensorFlow docs и tested build configurations имеет смысл, если:
- вам нужен старый TensorFlow;
- у вас нестандартная версия Python;
- у вас очень новая или необычная GPU;
- TensorFlow видит GPU, но во время реальной работы появляются:
  - `CUDA_ERROR_INVALID_PTX`;
  - `CUDA_ERROR_INVALID_HANDLE`;
  - предупреждения про PTX JIT и несовместимость с текущим `compute capability`.

Это и есть момент, когда простой student-flow заканчивается и начинается advanced compatibility reasoning.

## Практический Decision Tree
1. Нужен ли вам вообще локальный GPU.
2. Если нет, не занимайтесь подбором версий: используйте `auto`, `local-cpu`, `colab-gpu` или `kaggle-gpu`.
3. Если да, идите по blessed path: `Linux/WSL2 + tensorflow[and-cuda]`.
4. Если GPU не виден, это setup-проблема.
5. Если GPU виден, но real training падает, это уже compatibility-проблема.
6. На этом шаге нормальный учебный fallback:
   - `local-cpu`;
   - `colab-gpu`;
   - `kaggle-gpu`.
7. Только после этого имеет смысл разбираться с exact version matrix.

## Типовой Пограничный Случай
Полезный пример для мышления:
- у студента новая видеокарта;
- `nvidia-smi` работает;
- TensorFlow видит `GPU:0`;
- короткий `matmul` на GPU может даже выполниться;
- но полноценный notebook падает на реальных kernels.

Что здесь важно понять:
- это не обязательно ошибка в `RNN`, `Attention` или `Transformer`;
- это не значит, что студент "неправильно написал lab";
- это часто означает, что локальный TensorFlow GPU-стек ещё не очень дружит с конкретной картой.

Правильная реакция:
- не переписывать notebook вслепую;
- не заниматься бесконечным ручным твикингом архитектуры;
- сначала зафиксировать рабочий fallback на CPU или cloud GPU;
- потом уже, отдельно от лабораторной, разбирать совместимость версий.

## Продвинутый Appendix: Как Читать Official Docs
Если хотите понимать картину глубже, смотрите в TensorFlow docs две вещи:

### 1. Pip install docs
Они отвечают на вопрос:
- какой сейчас штатный способ установки;
- где TensorFlow рекомендует `tensorflow[and-cuda]`;
- почему для современных версий Windows native GPU не считается основным путём.

### 2. Tested build configurations
Они нужны не для первого запуска, а для нестандартных случаев:
- старые версии TensorFlow;
- unusual Python versions;
- сборка из source;
- ситуации, где нужно понять точную связку `TensorFlow -> Python -> CUDA -> cuDNN`.

Как это читать на практике:
- сначала смотрите версию TensorFlow;
- потом допустимые версии Python;
- потом какой GPU-path TensorFlow вообще поддерживает для этого сценария;
- только потом думаете про более низкий уровень совместимости.

## Что Должно Остаться У Студента После Этого Guide
- понимание, что у курса есть один нормальный локальный GPU-path;
- привычка сначала выбирать сценарий запуска, а не прыгать сразу в таблицы версий;
- умение различать setup-проблему и compatibility-проблему;
- спокойное отношение к fallback на `CPU`, `Colab` или `Kaggle`, если локальный GPU нестабилен;
- понимание, где заканчивается учебная задача и где начинается системная отладка GPU-стека.
