## Зачем эта лабораторная
Это не “игрушка ради игрушки”.
Она специально построена так, чтобы показать слабое место модели без позиции:
- один и тот же набор токенов;
- разные порядки;
- разные метки.

То есть студент буквально видит, зачем Transformer нужен positional embedding.

## Логика задачи
Правило метки:
- `y = 1`, если токен `7` встречается раньше токена `3`;
- `y = 0`, если токен `3` идёт раньше.

Гарантируется, что оба токена есть в каждой полезной последовательности.

Если смотреть только на набор токенов, примеры могут быть одинаковыми.
Различие создаёт именно порядок.

## Что нужно собрать в notebook
### 1. Генератор данных
Проверить, что:
- длина последовательности меняется;
- `PAD = 0`;
- `7` и `3` действительно вставляются;
- метка согласована с их позициями.

### 2. `TokenAndPositionEmbedding`
Проверить, что:
- токены идут во вход `(batch, time)`;
- на выходе получается `(batch, time, embed_dim)`;
- позиционный вектор добавляется к token embedding, а не заменяет его.

### 3. `TransformerEncoderBlock`
Минимальный контракт:
- `MultiHeadAttention`;
- residual;
- `LayerNormalization`;
- feed-forward;
- ещё одна residual + normalization;
- передача padding mask.

### 4. Classifier head
После encoder block нужен один вектор на объект.

Самый безопасный путь:
- сделать masked average pooling;
- затем `Dense`;
- затем финальный sigmoid для binary classification.

### 5. Trace attention
Полезный отдельный маршрут:
- взять embedding;
- прогнать через encoder block с `return_attention_scores=True`;
- усреднить score'ы по головам;
- показать heatmap на одном примере.

## Где чаще всего ломаются `TODO`
### Генерация данных
- забыли вставить оба специальных токена;
- label считается не по позиции, а по значению;
- padding перепутан с обычными token ids.

### Embedding
- позиционный индекс не совпадает с длиной последовательности;
- token embedding и position embedding имеют разный `embed_dim`;
- забыли вернуть mask дальше по сети.

### Attention
- ожидали форму `(batch, T, T)`, а пришла `(batch, heads, T, T)`;
- attention mask передали не той формы;
- padded позиции не скрыты.

### Pooling
- сделали обычное усреднение по всем `T`, включая `PAD`;
- потеряли разницу между реальными токенами и padded хвостом.

## Как читать результат
Хороший исход:
- `test accuracy >= 0.95`;
- на ручных примерах модель различает перестановки;
- attention карта не сосредоточена на padded хвосте;
- хотя бы часть heads реагирует на важные токены `7` и `3`.

Необязательно ждать “идеальной” карты.
Для учебной задачи достаточно, чтобы attention выглядел осмысленно, а не как случайный шум.

## Критерии завершения
`ЛР01` можно считать закрытой, если одновременно выполнено всё ниже:
- два ручных примера с перестановкой `7` и `3` дают разные предсказания;
- итоговая метрика держится на уровне `test_acc >= 0.95`;
- heatmap строится только по non-PAD части последовательности;
- студент может объяснить, зачем в этой задаче нужны и positional embedding, и padding mask.

## Порядок самопроверки
1. Взять два ручных примера с одним набором токенов и разными метками.
2. Проверить форму `X`, `mask`, `embeddings`.
3. Проверить форму `attention_scores`.
4. Проверить, что pooled-вектор считается только по полезным позициям.
5. Только потом смотреть на `accuracy`.

## Что должно остаться после ЛР
- positional embedding решает реальную проблему, а не добавляется “для красоты”;
- self-attention можно читать и проверять через trace;
- mask нужна не только в `seq2seq`, но и в Transformer encoder.
