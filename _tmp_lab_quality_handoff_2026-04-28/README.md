# Handoff: улучшение учебного качества лабораторных материалов

Эта временная директория предназначена для следующего агента, который будет
дорабатывать `edu-ai-math-essentials`. Здесь не меняются сами лабораторные
материалы: это карта улучшений, критериев и проверок, которые стоит перенести
в основной учебный контент отдельной задачей.

## Что уже видно по проекту

- В проекте есть сильная учебная структура: `themes/00-Foundations`,
  затем тематические блоки `01-RNN` ... `05-Full-Transformer`.
- В `AGENTS.md` уже закреплены важные правила: не перенумеровывать
  существующие `TODO`, держать student notebooks и solution notebooks
  синхронизированными 1:1, сохранять поток
  `contract -> theory -> manual example -> TODO -> checks -> diagnostics`.
- В `00-Foundations`, `01-RNN` и `02-Attention` уже много дружелюбных
  beginner-guide материалов.
- В темах `03-Transformer`, `04-Autoregression`, `05-Full-Transformer`
  заметны английские секции docstrings вида `Args:` и `Returns:`. Для
  русскоязычного курса лучше унифицировать их как `Аргументы:`,
  `Возвращает:`, `Исключения:`.
- В репозитории сейчас не обнаружена единая директория `scripts/` с
  contract-check скриптами по аналогии с соседним проектом
  `edu-operations-research`. Контракты в основном описаны текстом в README,
  guides и самих notebook-ах.

## Как читать эти рекомендации

1. Начать с `00_global_quality_contract.md`. Это общий стандарт качества для
   теории, guides, starter notebook-ов, solution notebook-ов, runtime-слоя,
   диагностик и отчетов.
2. Затем открыть `01_foundations_recommendations.md`. Там собраны улучшения
   для общего входного блока `themes/00-Foundations`.
3. После этого пройти `02_lab_by_lab_recommendations.md`. Там даны конкретные
   предложения для `01-RNN`, `02-Attention`, `03-Transformer`,
   `04-Autoregression`, `05-Full-Transformer`.
4. Только после согласования отдельной задачи переносить рекомендации в
   реальные материалы курса.

## Неграницы этой handoff-директории

- Не менять числовые, датасетные и runtime-постановки без отдельной задачи.
- Не переписывать notebook-и массово ради косметики, если это ломает
  историю прохождения.
- Не renumber существующие `TODO`.
- Не раскрывать student notebooks до уровня solution notebooks.
- Не запускать тяжелое обучение в обычном CI; тяжелые GPU-прогоны должны быть
  отдельными acceptance/profile сценариями.

## Рекомендуемый результат следующей задачи

После применения этих рекомендаций студент должен видеть единый учебный язык
во всех темах: сначала интуиция "на пальцах", затем строгий контракт данных,
затем читаемый код, затем проверки форм/масок/метрик, затем интерпретация
результата и требования к отчету. Машинные проверки должны фиксировать этот
стандарт, чтобы будущие правки не возвращали английские docstrings,
плотные one-line тензоры, рассинхрон starter/solution или скрытые изменения
датасетов.
