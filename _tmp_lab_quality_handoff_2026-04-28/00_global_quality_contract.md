# Единый контракт качества для учебных материалов

Этот документ описывает стандарт, который стоит закрепить для всех материалов:
`themes/00-Foundations` и тематических лабораторных блоков
`themes/01-RNN` ... `themes/05-Full-Transformer`.

## 1. Общие правила изменения материалов

- Числовые постановки, датасеты, фиксированные seeds, thresholds, budget gates
  и критерии прохождения не меняются без отдельной явно согласованной задачи.
- Существующие `TODO` не перенумеровываются. Если нужен новый шаг, добавлять
  его как локальное уточнение рядом с существующим блоком или завести отдельную
  задачу на аккуратную миграцию структуры.
- Для каждой пары `starter/solution` сохраняется 1:1 alignment:
  одинаковые крупные разделы, одинаковые контрольные точки, одинаковая
  терминология, одинаковая последовательность вычислений.
- Student notebooks остаются учебными заготовками. Они должны содержать
  guided-пропуски, подсказки, `assert`, preview-таблицы и диагностические
  мини-проверки, но не должны превращаться в полностью готовое решение.
- Solution notebooks могут содержать полную реализацию, но должны быть
  написаны так, чтобы студент мог читать их как академический разбор, а не как
  плотный служебный скрипт.

## 2. Теория и guides

Каждый теоретический блок должен иметь три слоя:

1. Интуитивный слой.
   Объяснение "на пальцах": какая бытовая или прикладная ситуация стоит за
   моделью, что выбирает модель, что считается ошибкой, что значит "хороший"
   результат.

2. Формальный слой.
   Четкие обозначения тензоров, размерностей, масок, loss-функций, метрик и
   ограничений. Важно не только назвать объект, но и указать его форму,
   смысл осей и единицы интерпретации.

3. Операционный слой.
   Пошаговая последовательность решения: подготовить данные, проверить формы,
   построить модель, выполнить ручной пример, обучить, проверить метрики,
   диагностировать ошибки, интерпретировать результат в отчете.

Для сложных тем полезен повторяемый раздел:

```text
Как читать эту тему новичку
```

В нем нужно коротко объяснить:

- какие прежние темы нужны;
- какие формы тензоров надо держать в голове;
- какие ошибки чаще всего возникают;
- какие sanity-checks должны сработать до обучения;
- какой результат считается достаточным для учебной цели.

## 3. Notebook-код

Код должен быть учебным, читаемым и проверяемым:

- имена функций и переменных можно оставлять английскими;
- учебные комментарии в code cells лучше писать по-русски;
- function docstrings в notebook-ах делать русскими Google-style:

```python
def make_dataset(...):
    """Создает учебный набор последовательностей.

    Аргументы:
        ...: ...

    Возвращает:
        ...: ...

    Исключения:
        ...: ...
    """
```

- английские секции `Args:`, `Returns:`, `Raises:` для русскоязычного курса
  стоит заменить на `Аргументы:`, `Возвращает:`, `Исключения:`;
- кодовые матрицы, векторы и тензорные примеры оформлять многострочно;
- между логическими блоками кода оставлять пустые строки;
- избегать скрытой магии в длинных one-liner выражениях;
- для сложных операций использовать промежуточные имена:
  `token_embeddings`, `padding_mask`, `causal_mask`, `decoder_target`,
  `attention_scores`, `logits`, `loss_value`.

Пример читаемого форматирования:

```python
toy_tokens = np.array(
    [
        [SOS_ID, 4, 7, EOS_ID, PAD_ID],
        [SOS_ID, 3, 1, 8, EOS_ID],
    ],
    dtype=np.int32,
)

expected_shape = (
    batch_size,
    sequence_length,
    embedding_dim,
)
```

## 4. Контракт данных

Для каждой лабораторной должен быть явный data-contract:

- что такое один объект выборки;
- какие входы получает модель;
- какие targets она предсказывает;
- какие формы имеют все ключевые массивы;
- где появляются `PAD`, `SOS`, `EOS`;
- где и зачем применяется mask;
- какие индексы запрещено смешивать.

Для sequence/Transformer тем особенно важно фиксировать:

- `encoder_input`;
- `decoder_input`;
- `decoder_target`;
- `padding_mask`;
- `causal_mask`;
- `cross_attention_mask`;
- `attention_scores`;
- ожидаемые batch/time/channel оси.

## 5. Starter/solution alignment

Для каждой пары notebook-ов рекомендуется автоматическая проверка:

- starter и solution имеют одинаковые markdown-заголовки верхнего уровня;
- starter содержит `TODO`, solution не содержит нерешенных `TODO`;
- solution содержит все helper-функции starter-а;
- student notebook не содержит полностью исполняемый готовый solve/train блок
  там, где должен быть учебный пропуск;
- checkpoints, assertions, diagnostics и report checklist расположены в
  одинаковом порядке.

## 6. Outputs и execution policy

- Легкие toy solution notebooks можно исполнять в обычной проверке.
- Student notebooks обычно валидировать структурно, без требования полного
  исполнения.
- Тяжелые CPU/GPU notebook-и не должны запускаться в обычном CI полностью.
  Для них нужны отдельные режимы:
  `structure-check`, `smoke-check`, `profile-run`, `gpu-acceptance-run`.
- Сгенерированные outputs хранить только там, где они реально помогают
  учебному чтению или acceptance-доказательству.
- Побочные rewrites notebook metadata и outputs вне изменяемой темы
  откатывать.

## 7. Отчет студента

В каждом README или notebook-финале нужен короткий раздел "Что должно быть в
отчете". Минимальный набор:

- формулировка задачи простыми словами;
- data-contract с ключевыми формами;
- таблица или список важных hyperparameters;
- sanity-checks до обучения;
- итоговые метрики;
- пример предсказания/генерации;
- диагностика типичной ошибки;
- интерпретация attention/mask/hidden states там, где это относится к теме;
- вывод: что получилось, что не получилось, почему результат можно считать
  корректным или недостаточным.

## 8. Suggested Check Scripts

Рекомендуется добавить директорию `scripts/` и закрепить проверки:

- `scripts/validate_notebooks.py` — чтение всех notebook-ов, JSON-валидность,
  обязательные cell ids, отсутствие битых metadata-полей, базовая структура.
- `scripts/check_notebook_contracts.py` — общие контракты:
  starter/solution alignment, наличие `TODO` только в student notebook-ах,
  запрет грубых PEP-нарушений, отсутствие tabs, лимит длинных строк.
- `scripts/check_lab_quality_contracts.py` — тематические правила качества:
  русские docstrings, русские учебные комментарии, многострочные матрицы и
  тензоры, обязательные data-contract markers, report checklist markers.
- Специализированные runtime/GPU checks — отдельные скрипты для `04` и `05`,
  которые проверяют preflight, наличие budget gates, leakage checks,
  perplexity/baseline/generation gates, но не запускают тяжелое обучение в
  обычном CI.

Для каждого check-script стоит явно фиксировать список файлов, которые он
проверяет. Если notebook переименован или добавлен, проверка должна падать,
пока контракт не обновлен сознательно.

## 9. CI и локальный маршрут проверки

Минимальный локальный маршрут после учебных правок:

```bash
python3 scripts/validate_notebooks.py
python3 scripts/check_notebook_contracts.py
python3 scripts/check_lab_quality_contracts.py
git diff --check
```

Для тяжелых тем:

```bash
python3 scripts/check_runtime_contracts.py
python3 scripts/check_gpu_notebook_contracts.py
```

Эти проверки должны подтверждать структуру и контракты, а не выполнять
дорогое обучение по умолчанию.
